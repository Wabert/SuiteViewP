When to run what

While editing: focused tests Run the exact test or test file beside the changed behavior. Usually 1-10 seconds.
After a logical change: affected modules Run every touched subsystem's test files. The recent five-module check ran 134 cases in 12 seconds.
Before declaring code ready: default regression Run all 768 default cases. Current measured time is 1:58.
For cross-component changes: integration Add the 170 offline integration cases. Current measured time is 0:07.
Before releases or when relevant: external/deep lanes Run live DB2, Outlook, and performance lanes only in the environment they require.
Everything, explicitly: 
venv\Scripts\python.exe -m pytest -o addopts="" tests\ -q
This restores all 955 cases, including live and long-running checks.

What these tests do - and do not do

Strong coverage

  • Actuarial and insurance-domain calculations
  • GLP, GSP, TAMRA, premium, loan, withdrawal, and force-out rules
  • Illustration reports, ledgers, saved cases, and input compilation
  • Audit SQL generation and query-object/DataForge persistence
  • PyQt control state, signals, selection, and data presentation
  • Rate workup conversion and validation

Important limits

  • Offscreen Qt checks behavior, not every visual spacing or color detail
  • Live DB2 tests require the work environment and should not silently fall back
  • Outlook tests depend on the installed desktop application and active mailbox
  • Legacy diagnostics may verify connectivity more than precise assertions
  • Visual changes still need the desktop screenshot workflow when appearance matters
